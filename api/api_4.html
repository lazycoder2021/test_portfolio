Anatomy of a REST API

Going back to the open weather api example we used to 'get' weather data of a city, our URL looked like this

Let's break down this URL to understand REST API URL conventions.

A REST API URL consists of:

  • URL - that part of the url which remains unchanged for all api requests. Usually a sub domain of the comapany reserved specifically for handing out the apis. In most cases, it ends with a .com or .org (like sub domains do) but in some cases, it ends with the api version number (like in the case of open wether api).
  • Endpoint - that part of the url before the '?' mark but after the base url. Endpoint bascially lays down the path for accessing the resource.
  • Resource - the information being accessed. In this case, it's the weather. Usually indicated or hinted by the endpoint itself.
  • Query Parameters - the part of the url after the '?' mark written in key: value pairs (q=london).
  • API KEY or ID - a unique key or ID which authenticates the api subscriber (this is also a query parameter indicated by the 'appid' key.)

These parts put together make up the REST API request. And what you get back on the screen when the url gets executed is the raw response which is later given a UI and presented to the client. In fact, you also have a number of UI based tools to make requests that make the lives of developers easier as they build and test the apis. POSTMAN is one such popular tool.
